The images clearly confirm the problem. The 16MP capture  shows a clean, properly exposed image with no color artifacts. The 64MP capture  has severe **horizontal magenta banding** covering the entire frame — every row is tinted with alternating magenta/dark lines at a fixed frequency.[1][2]

## What the Banding Pattern Tells You

The **horizontal line pattern** (not random noise, but periodic rows) is the key diagnostic clue. This is almost never a white balance or AWB issue — it is a **black level / pixel readout timing issue** specific to the full-resolution mode. In the OV64A40, full 64MP uses a different readout path than the 4×4 binned 16MP mode, and the black level subtraction must be recalibrated for that path.

## Most Likely Cause: BLC (Black Level Correction) Mismatch

In the 64MP mode, the sensor reads all rows without pixel binning. The black reference rows and their subtraction coefficients differ from binned mode. If BLC is not properly initialized for the full-res mode, **every other Gr/Gb row gets a wrong offset**, which after demosaicing produces the characteristic horizontal magenta stripes (red+blue elevated, green suppressed on alternating rows).

## What to Fix in `ov64a40.c`

### 1. Enable Auto BLC for 64MP mode

In your 64MP-specific register table, add or verify:

```c
{ CCI_REG8(0x4000), 0x27 }, /* BLC: enable auto, apply to all channels */
{ CCI_REG8(0x4001), 0x03 }, /* BLC start line */
{ CCI_REG8(0x4002), 0x45 }, /* BLC auto mode */
```

The value `0x27` enables automatic black level calibration. If this was previously `0x00` or missing from the 64MP table, BLC is off and the raw pixel values are uncorrected.

### 2. Set Per-Channel Black Level Offsets

The registers `0x4030`–`0x4037` are the per-channel BLC offsets (R, Gr, Gb, B × 2 bytes each). For the 64MP mode, try:

```c
{ CCI_REG8(0x4030), 0x00 },
{ CCI_REG8(0x4031), 0x10 }, /* R offset */
{ CCI_REG8(0x4032), 0x00 },
{ CCI_REG8(0x4033), 0x10 }, /* Gr offset */
{ CCI_REG8(0x4034), 0x00 },
{ CCI_REG8(0x4035), 0x10 }, /* Gb offset */
{ CCI_REG8(0x4036), 0x00 },
{ CCI_REG8(0x4037), 0x10 }, /* B offset */
```

Start with equal values (`0x10`) across all channels. If the banding shifts color but doesn't disappear, the Gr/Gb rows have a **different offset than R/B** — try bumping `0x4033`/`0x4035` (the green channel offsets) higher until the magenta neutralizes.

### 3. Check `0x3208` Group Hold

The banding can also occur if register writes during mode switch are **not grouped/held atomically**. Wrap your 64MP-specific register table writes in a group hold:

```c
{ CCI_REG8(0x3208), 0x00 }, /* Group hold start */
/* ... all 64MP registers ... */
{ CCI_REG8(0x3208), 0x10 }, /* Group hold end */
{ CCI_REG8(0x3208), 0xa0 }, /* Group hold launch */
```

This ensures the sensor doesn't start a partial frame mid-configuration.

### 4. Verify `0x4501` (BLC target)

```c
{ CCI_REG8(0x4501), 0x00 }, /* BLC target level */
```

A non-zero target here shifts the black point and causes banding if it was calibrated for the binned mode.

## Quick Test

As a fast sanity check, temporarily write `0x4000 = 0x27` at runtime via `i2cset` while the 64MP stream is running, and see if the banding changes or disappears. If it does, the fix is confirmed to be a BLC configuration issue in the 64MP register table.
